# TASK-005 · authoring 原子成本塌陷窗口证据

## 为什么要这个任务

TASK-004 的墓碑（HyperCard / Access / FileMaker / Pipes / App Maker）已证「谁试过、为何死/退守」，唯独缺「窗口变了什么」——即：生成式 AI 是否真把 authoring 原子成本从「学会工具」塌到「描述任务」。没有这条，CAND-004 三件套不全，不立档。

## 命门（Claude 裁决口径，取证须服务于此）

no-code 当年也号称降门槛，结果是门槛**位移**（写代码 → 会 authoring + 会维护 + 会验收），不是消失——Access/FileMaker 的退守正死于此。故本任务必须分别量两个门槛：

- authoring 门槛（造出来）：大概率在塌。
- 验收门槛（判断对不对）：**命门**。塌了才是真颠覆（连「你看一下对不对」都省）；没塌则 CAND-004 证据封顶 B，且须标注「门槛位移而非消失」。

注意：验收门槛正是前三候选「你看一下对不对」的同一隐含元假设，若它没塌，说明四候选共享同一未质疑前提，本爆的「换沟」要打折。

## 两条防漂移铁律

1. **不是扫「有哪些 AI 造软件的产品 / 它们能做什么」**。这是限制#1·AI 中心框架，会退化成功能清单。本任务量的是 authoring 原子成本结构是否塌陷——证据形态必须是**成本/门槛结构**，不是玩家名单。凡以「某产品能做 X」充证据的行，判无效。
2. **两个门槛分开量，禁止混谈**：authoring 门槛（把软件造出来）与验收门槛（判断造出来的对不对）各自独立取证。验收门槛没塌就必须显性标注「门槛位移而非消失」。
